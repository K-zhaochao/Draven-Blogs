---
title: 6.分布式并发控制（分布式锁）
---
### 1. 核心原理

利用 Redis 的 `SET key value NX EX time` 原子命令。通过 **Redisson** 框架的“看门狗”机制防止业务超时锁提前自动释放，以及通过“判断锁的标识”防止错删他人的锁。

### 2. Java 代码实现

虽然我们可以自己用 `stringRedisTemplate.opsForValue().setIfAbsent()` 手写，但在生产环境中，**强烈推荐直接使用成熟的 Redisson 框架**，因为它完美解决了你我讨论的“锁续命”和“误删”问题。（生产环境推荐用 Redisson）

**第一步：引入依赖**
``` xml
<dependency>
    <groupId>org.redisson</groupId>
    <artifactId>redisson-spring-boot-starter</artifactId>
    <version>3.23.1</version> </dependency>
```

**第二步：核心业务代码**
``` java
@Autowired
private RedissonClient redissonClient;

public void secureSecKill(Long dishId) {
    // 1. 获取锁对象（指定锁的 Key 范围）
    RLock lock = redissonClient.getLock("lock:dish:" + dishId);
    
    // 2. 尝试获取锁
    // tryLock 参数：(最大等待时间, 锁自动释放时间, 时间单位)
    // 💡注意：如果不传“锁自动释放时间”，Redisson 才会开启【看门狗机制】自动续命！
    boolean isLock = lock.tryLock(); 
    
    if (!isLock) {
        // 拿锁失败，说明有人在抢，直接返回失败或重试
        throw new RuntimeException("服务器繁忙，请稍后再试！");
    }

    try {
        // 3. 拿到锁成功，执行核心业务（扣减 MySQL 库存）
        deductStockFromMySQL(dishId);
    } finally {
        // 4. 💡核心注意点：释放锁，必须放在 finally 块中确保一定会执行
        // 并且要判断【锁是不是我持有的】以及【锁是不是还活着】
        if (lock.isHeldByCurrentThread()) {
            lock.unlock();
        }
    }
}
```

### 注意点：

- **释放锁的严谨性：** 释放锁必须写在 `finally` 块里！否则如果 `deductStockFromMySQL` 抛出异常，代码直接中断，`unlock()` 没执行，那又变成死锁了。
    
- **看门狗的触发条件：** 调用 `tryLock` 时，**千万不要**手动传第三个代表锁自动释放时间的数字。一旦你指定了数字（比如10秒），看门狗就**失效**了，锁到了10秒硬性释放。

# 高并发缓存防护（穿透与击穿）

### 1. 核心原理

- **防穿透**：MySQL 查不到，就在 Redis 塞一个 `null` 值并设置超短过期时间。
    
- **防击穿**：热点 Key 失效时，用互斥锁（如上面的分布式锁）只放一个线程去查数据库。
    

### 2. Java 代码实现（防穿透：缓存空对象）

这是对你“苍穹外卖”项目商品查询逻辑的终极严谨升级：
``` java
public Dish getDishWithDefend(Long id) {
    String key = "cache:dish:" + id;
    
    // 1. 从 Redis 查询
    String json = stringRedisTemplate.opsForValue().get(key);
    
    // 2. 判断是否存在
    if (StrUtil.isNotBlank(json)) {
        // 命中，直接转成对象返回
        return JSONUtil.toBean(json, Dish.class);
    }
    
    // 3. 💡防穿透核心：判断命中的是不是我们故意塞进去的那个 "null 空字符串"
    if (json != null) {
        // 走到这里，说明 json 是 ""（空字符串），不是真正的 null（没数据）
        // 证明这是黑客以前攻击过的无效 ID，直接拒绝，不再查数据库！
        return null; 
    }

    // 4. 缓存未命中，且不是空字符串，真正去查 MySQL
    Dish dish = dishMapper.selectById(id);
    
    // 5. 数据库中也不存在
    if (dish == null) {
        // 💡防穿透核心：向 Redis 写入一个空字符串 ""，设置 2 分钟极短过期时间
        stringRedisTemplate.opsForValue().set(key, "", 2, TimeUnit.MINUTES);
        return null;
    }

    // 6. 数据库存在，正常写入缓存
    stringRedisTemplate.opsForValue().set(key, JSONUtil.toJsonStr(dish), 30, TimeUnit.MINUTES);
    return dish;
}
```

### 注意点：

- **分清 `null` 和 `""`**：在 Java 中，从 Redis 拿不到数据返回的是真正的 `null`。而我们为了防穿透，塞进去的是一个有长度的空字符串 `""`。在代码的第 3 步，必须用 `json != null` 把这个 `""` 拦下来，否则它又漏过去查数据库了。

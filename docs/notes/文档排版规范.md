# 文档排版规范

为了保证本知识库（Draven_Note）的整洁度、可读性和一致性，特建立以下文档排版与书写规范。请在后续记录和整理笔记时，尽量遵循本规范。
![img](/Draven_Note_Images/Pasted image 20260509005725.png)
## 1. 标题规范 (Headers)

- **层级分明**：使用 `#` 来表示不同层级的标题，确保标题架构拥有清晰的逻辑结构。
- **空格要求**：在 `#` 号与标题文本之间，**必须**保留一个空格。例如：`## 1. 概述`，避免写成 `##1. 概述`。
- **避免过度嵌套**：通常情况下，标题层级建议不要超过 4 级（即最多使用 `####`），过多的层级会导致内容结构显得冗杂。

## 2. 文本排版与间距 (Spacing)

- **段落分隔**：不同的段落之间应保留**一个空行**。不要使用多个连续空行。
- **中英文排版**：在中文字符和英文字符（或数字）之间，推荐保留一个半角空格，这能显著提升阅读体验。例如：`这是一个 Java 程序` 而不是 `这是一个Java程序`。
- **列表使用**：
  - 无序列表使用 `-`、`+` 或 `*`，符号后必须加一个空格。
  - 有序列表使用 `1.`、`2.` 等，点号后必须加一个空格。
  - 列表项之间如果不包含多行内容，可以不留空行；如果包含，可以保留一个空行以增加可读性。

## 3. 代码与高亮规范 (Code & Highlighting)

- **多行代码（代码块）**：
  - 在内容中包含多行代码、控制台输出、配置信息（如 JSON, XML, YAML）时，**必须**使用三个反引号（```` ``` ````）包裹形成代码块。
  - 必须为代码块指定对应的语言标签，以便于实现正确的语法高亮。例如：```` ```java ```` 或 ```` ```json ````。
  - **不要使用块引用 (`>`) 或单纯的文本缩进来代替代码块**，这会导致代码丢失高亮且不易复制。
- **行内代码**：
  - 针对正文中的类名、方法名、变量名、关键字或极其简短的一句代码，请使用单个反引号包裹。例如：`System.out.println()` 或 `public` 关键字。
  - 不要将多达几行或过长的单行代码强制使用带有 ` 的行内代码表示，应当拆分为代码块。

## 4. 引用与强调 (Blockquotes & Emphasis)

- **引用区块**：使用 `>` 用来引用摘抄的文字、官方文档、或者作为明显的“提示/Warning”。引用区块内也可嵌套其他排版格式。
- **粗体与斜体**：
  - 使用 **`文字`** 表示**粗体**，用于强调核心主旨或重要概念。
  - 使用 `*文字*` 表示*斜体*，用于辅助性的强调或术语解释。
  - 避免在一大段文字上乱用粗体，这会丧失重点。

## 5. 图片与链接 (Images & Links)

- **路径规范**：所有图片必须使用相对于该 Markdown 文件的正确路径。例如在子目录下应正确使用 `../` 跳出层级，并指向 `Draven_Note_Images` 统一图床。
- **图片命名**：截图保存时应具备有意义的名称，避免使用 `image-12345.png` 这类无意义名称（可根据笔记软件的插件自动重命名功能规范化）。
- **链接描述**：添加外部参考链接时，使用 `[链接描述](URL)` 格式，描述应准确反映目标内容。

## 6. 统一的结构 (Standard Structure)

针对某些特定主题的笔记（如框架总结、API使用等），建议采用以下默认的内容架构：

1. **概述/简介**（解释概念）
2. **核心原理**（图解等辅助说明）
3. **代码示例**（标准的示例代码块）
4. **注意事项/踩坑总结**（使用块引用或列表）
